Level 1｜基礎應用（0–1M）

會用 AI 產出文件與查詢

  • 用 AI 產出流程圖步驟與圖例（Draw.io）
  • 從需求句子生成 SQL，並優化索引/條件
  • 生成規格書、簡報(opetional)，套用結構模板

完成標準：可獨立依需求完成文件/SQL/規格書or簡報。

Level 2｜半自動化（2–4M）

能將 AI 產出整合至前端

  • 依規格由 AI 生成G2頁面的 HTML/CSS/JS 雛形
  • 串接 Mock API / 真實 API(已有現成的話)
  • 建立樣式規範與元件化（表單、表格）

完成標準：可運行的 Demo，與規格符合度 ≥85%。

Level 3｜AI 協作開發（4–6M）

能交付 AI Bot 與 BI 儀表板

  • 使用 Azure OpenAI 建立 一處/專案/特定系統 Bot
  • 使用 Azure OpenAI 建立 AI Agent(otpional)
  • 打造 Superset 儀表板與資料模型(配合數據中台建置)
  • 設計評估：Bot 命中率、BI 指標正確率

完成標準：可用 Bot or AI Agent(otpional)+ 1 份正式 BI報表

Level 1 任務包 ＋ Prompt 範本庫

適用對象：系統規劃/分析人員（第 0–4 週）

任務包內容（交付物）
  • 流程圖：1 份核心流程（包含泳道與例外分支），檔案格式 .drawio/.png
  • SQL：3 條查詢（CRUD 至少各 1），含索引/Explain 分析與優化建議
  • 簡報：10 頁提案/教學簡報（封面/目錄/痛點/方案/流程/資料設計/風險/時程/KPI/結語）
  • 提交說明：需求摘要、使用的 Prompt、AI 產出後的人工作業清單
Prompt：流程圖（Draw.io） 
你是資深系統分析師，請根據以下需求生成「流程步驟清單」與「泳道圖元素清單」，用標準 BPMN 名詞，並附上圖例建議。
【專案名稱】{{project_name}}
【對象/泳道】{{swimlanes}}（例如：使用者、前端、後端、DB、第三方）
【需求摘要】{{requirements}}
輸出格式：
1) 流程步驟（序號＋動詞開頭＋條件分支）
2) 泳道圖元素（節點類型、名稱、說明、連線關係）
3) Draw.io 圖層規劃與配色建議

小技巧：請 AI 先產「元素清單」再手動或用外掛導入 Draw.io。

Prompt：SQL 產出與優化 
你是資料庫設計師。請依需求產出 SQL（以 {{db_type}} 為主，例如 SQL Server / PostgreSQL），並提供索引與 Explain 優化建議。
【資料表結構】{{tables_schema}}
【需求敘述】{{query_requirements}}
請輸出：
A. 基礎查詢 SQL（含註解）
B. 可能的索引（含欄位順序與理由）
C. Explain/EXPLAIN ANALYZE 觀察重點與替代寫法
D. 邊界情境測試資料（5 筆）

評估：可執行、成本可被 Explain 驗證、並覆蓋需求邊界。

Prompt：資料表設計（正規化＋命名） 
你是資料模型師。請依需求產出實體關係（ER）清單、命名規範與主外鍵設計，並給出建立表格的 SQL。
【需求摘要】{{requirements}}
請輸出：
1) 實體與屬性（英文小駝峰命名、欄位型別）
2) 主鍵/外鍵與唯一性約束
3) 正規化說明（達成 3NF 並說明例外）
4) DDL（CREATE TABLE …）與索引
Prompt：規格書生成 
你是「資深系統分析師＋規格書編寫教練」。請依下列輸入，產出一份「可交付的系統規格書（SRS）」，要求：以繁體中文、Markdown 排版、章節完備、表格清晰、可直接交付研發/測試/上線使用。
【系統名稱】{{system_name}}
【文件用途】{{doc_purpose}}（例如：對外發包/內部開發/驗收依據）
【受眾】{{audience}}（例如：PM、SA、Developer、DBA、QA、Trainer）
【範疇】{{scope}}
【排除範疇】{{out_of_scope}}
【利害關係人】{{stakeholders}}
【合規/稽核】{{compliance}}（例如：資安、個資、ISO、法規）
【命名規範】{{naming_rules}}（例如：資料表/欄位/API 命名）
【資料庫類型】{{db_type}}（例如：SQL Server / PostgreSQL）
【現況痛點】{{pain_points}}
【需求摘要】{{requirements}}
【使用者角色/Persona】{{personas}}
【上下游整合/介接系統】{{integrations}}
【非功能性目標】{{nfr_targets}}（效能、可用性、容量、SLA）
【時程與里程碑】{{timeline}}
【KPI/驗收指標】{{kpi}}
【素材/附件】{{materials}}
=== 請輸出以下章節（Markdown） ===
1. 封面與修訂史  
   - 文件版本、日期、作者、審核人、修訂紀錄（表格）

2. 摘要與目標  
   - 系統願景、商業目標、成功衡量（KPI 對應）

3. 名詞定義與縮寫  
   - Domain 名詞、縮寫說明（表格：詞彙/定義/備註）

4. 系統範疇與邊界  
   - In/Out of Scope 條列、上下游系統關係文字描述（可附簡圖說明）

5. 使用者角色與權限矩陣  
   - 角色列表、責任、權限（表格：Role/功能/CRUD/備註）

6. 需求分解（BRD → FRD）  
   - 商業需求（BR）→ 功能需求（FR）對照表  
   - 業務規則（Business Rules）條列

7. 業務流程（BPMN 文字化）  
   - 以「步驟清單＋泳道元素清單」描述流程，含例外/邊界情境  
   - 格式：
     - 流程步驟：編號、敘述（動詞開頭）、分支條件
     - 泳道元素：節點類型/名稱/說明/前後關係
   - （若需圖檔：請附 draw.io 圖層規劃與節點列表）

8. 介面/頁面規格（UI Spec）  
   - 以頁面為單位列出：目的、使用者動線、元件  
   - 表格：欄位/型別/長度/格式/驗證規則/預設值/是否必填/提示文/錯誤訊息

9. 資料模型與資料字典  
   - 實體（表）清單、主外鍵關係、正規化說明（達成 3NF 的理由/例外）  
   - 資料字典表格：表名/欄位/型別/PK/FK/唯一/允空/描述  
   - 產出對應 {{db_type}} 的 DDL（`CREATE TABLE`＋索引），放在 ```sql 區塊

10. API 規格（REST/GraphQL 擇一或並列）  
    - 路徑/方法/用途、Request Params、Request/Response JSON Schema、狀態碼與錯誤碼  
    - 範例：以 ```json 區塊提供

11. 整合介面與事件（MQ/Webhook/ETL）  
    - Topic/Queue/事件結構、重試/去重/冪等性策略、時序與一致性  
    - 資料交換頻率、批次窗口、失敗補償流程（文字步驟）

12. 非功能性需求（NFR）  
    - 效能（TP95、QPS、批次 SLA）、容量規劃、可用性、備援、高可用/容錯  
    - 監控 KQI/KPI、告警閾值與處置SOP

13. 安全與合規  
    - RBAC 權限模型、資料分級與遮罩、審計 Log 欄位與保留政策  
    - 傳輸/靜態加密、密鑰輪替、隱私/個資合規要點

14. 驗收準則與測試案例  
    - 每個功能的驗收條件以 Gherkin（Given-When-Then）撰寫  
    - UAT 測試案例表、邊界/例外測試、資料品質校驗

15. 部署與遷移計畫  
    - 環境（Dev/Test/Staging/Prod）、版本控管、滾動/藍綠/金絲雀發布  
    - 資料遷移步驟、回滾策略、相依服務切換方案

16. 風險與緩解  
    - 風險清單（表格：風險/等級/影響/機率/緩解/備援）

17. PoC（如適用）  
    - PoC 範圍、成功門檻、時間盒、驗收指標、銜接正式版路徑

18. 附錄  
    - 命名規範、畫面元件庫、Prompt 範本、參考文件連結

=== 輸出規範 ===
- 全文使用繁體中文。  
- 以 Markdown 呈現所有章節；清單與表格必要且完整。  
- SQL 與 JSON 放入對應語言的三引號程式碼區塊。  
- 不可憑空捏造未知資訊；遇缺資料以「TODO/待確認」列點，並在文末輸出「需求澄清清單」。  
- 於文末輸出「品質檢查清單 (Checklist)」：可追蹤性、完整性、一致性、可測試性、安全性覆蓋、NFR 覆蓋。

可要求輸出為可下載的word檔格式
